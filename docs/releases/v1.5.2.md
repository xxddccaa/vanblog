# v1.5.2

`v1.5.2` 是一次偏后台编辑器暗色视觉收口的补丁版本，重点把 ByteMD 编辑区里那层不必要的额外深色内层背景去掉，让真实输入区域和外层编辑器面板重新保持统一。

如果只用一句话概括这个版本：

> `v1.5.2` 收掉了后台暗色模式下 ByteMD 编辑区里多出来的一层深色底板，让输入画布更干净，编辑和预览的视觉层级也更一致。

## 本版摘要

这次正式版本主要集中在三个方向：

- 暗色模式下 ByteMD 编辑区内部的 `CodeMirror` 画布改回透明，不再叠加一层额外深色背景
- 外层编辑器面板的暗色视觉继续保留，工具栏、分栏结构和预览区不受影响
- 发布文档、部署示例和自动化校验统一推进到 `v1.5.2`，确保镜像标签与 GitHub Release 口径一致

## 相比 v1.5.1 的变化

如果你已经在使用 `v1.5.1`，这一版不改默认编辑器，也不改编辑器切换逻辑；它主要修的是暗色模式下的视觉细节。

问题表现在这些场景里最明显：

- 管理后台切到暗色模式后打开文章编辑页
- ByteMD 左侧输入区出现一块比外层容器更深的内层背景
- 编辑代码块、公式或普通正文时，输入画布看起来像“又套了一层盒子”

`v1.5.2` 现在把这层多余背景去掉后，编辑区只保留外层面板的暗色基底，不再额外制造视觉分裂。

### 1. ByteMD 编辑画布不再叠加第二层深色背景

这次补丁针对的是 ByteMD 暗色模式下内部 `CodeMirror` 区域的背景继承关系：

- 外层 `bytemd-body` 继续负责整体暗色画布氛围
- 内层 `bytemd-editor`、`CodeMirror`、`CodeMirror-scroll`、`CodeMirror-lines` 改成透明
- 实际输入区不再单独绘制第二层深色背景

这样处理后，编辑区会更像一整块连续画布，而不是“外层面板 + 内层深色补丁”的组合。

### 2. 不影响已有暗色主题可读性

`v1.5.2` 这次只收口背景层，不改变这些关键点：

- 文本颜色仍然保持当前暗色主题对比度
- 光标、选区、边框和代码高亮规则继续沿用现有暗色变量
- 预览区、工具栏、状态栏和 Mermaid / 代码块样式不需要额外迁移

所以这版属于纯视觉收口，不是主题体系重做。

### 3. 镜像部署与文档基线同步推进到 v1.5.2

除了样式本身，这一版也把正式发版相关口径统一推进到 `v1.5.2`：

- 根目录版本号切到 `1.5.2`
- `README.md`、`RELEASE.md`、`DEPLOY.md` 的示例标签和版本说明同步更新
- 发布说明索引与部署校验测试一起指向新的 `docs/releases/v1.5.2.md`

这样后续继续用 `pnpm release:publish` 或锁版部署时，不会把样式修复和上一版说明混在一起。

## 主要特性 / 修复

### 后台编辑器暗色视觉

- ByteMD 输入区内部背景改为透明继承
- 去掉暗色模式下编辑区里不必要的深色内层底板
- 编辑区与预览区的整体视觉层级更统一

### 兼容性

- 不改变默认编辑器类型
- 不改变 `vanblog_editor_engine` 本地存储行为
- 不新增环境变量，也不需要数据迁移

### 发布与部署口径

- 仓库版本基线推进到 `v1.5.2`
- 发布说明索引补充 `v1.5.2` 条目
- 自动化部署文档校验同步覆盖新版本说明

## 部署与兼容性说明

`v1.5.2` 没有改变下面这些发布 / 部署边界：

- 默认公开拓扑仍然只有 `caddy`、`server`、`website`、`admin`、`waline`、`postgres`、`redis`
- AI 工作台仍然按需叠加 `docker-compose.ai-qa.yml` / `docker-compose.fastgpt.yml`
- bundled FastGPT 仍然不是默认公开拓扑的一部分
- `pnpm release:images` / `pnpm release:publish` 仍然只发布 VanBlog 核心 5 个镜像

这意味着：

- 不需要修改现有 compose 编排
- 不需要新增环境变量
- 已经在使用 `v1.5.1` 的部署可以直接升级到 `v1.5.2`

## docker compose 配置单

推荐正式部署仍然使用：

- `docker-compose.image.yml`
- `.env.release.example`

核心镜像示例保持：

```env
VANBLOG_DOCKER_REPO=kevinchina/deeplearning
VANBLOG_RELEASE_SUFFIX=v1.5.2-<image-id>
```

主栈启动：

```bash
docker compose -f docker-compose.image.yml up -d
```

只连接已有私有 FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml up -d
```

同时启用 bundled FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml -f docker-compose.fastgpt.yml up -d
```

如果只是 latest 快速体验，也可以继续使用：

```bash
docker compose -f docker-compose.latest.ai.yml up -d
```

## 升级步骤

1. 拉取 `v1.5.2` 对应代码或镜像
2. 确认 `.env` 中的 `VANBLOG_DOCKER_REPO` 与 `VANBLOG_RELEASE_SUFFIX`
3. 执行 `pnpm test:deploy`
4. 执行 `docker compose -f docker-compose.image.yml pull`
5. 执行 `docker compose -f docker-compose.image.yml up -d`
6. 打开 `/admin/editor`，在暗色模式下检查 ByteMD 左侧输入区是否仍出现多余的深色内层底板
7. 确认输入区背景和外层编辑器面板已经自然衔接，不再像嵌套了第二层深色盒子
8. 如需重复验收，建议顺手强刷浏览器资源，避免旧 CSS 缓存影响判断

## 回滚方法

如果线上有问题，把 `VANBLOG_RELEASE_SUFFIX` 改回上一版即可，例如：

```env
VANBLOG_RELEASE_SUFFIX=v1.5.1-<image-id>
```

然后重新执行：

```bash
docker compose -f docker-compose.image.yml up -d
```

如果只是个别浏览器继续显示旧背景，也可以先清理缓存或强刷页面，不影响 AI overlay 与 `/admin/ai` 的现有部署方式。
