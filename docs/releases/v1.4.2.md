# v1.4.2

`v1.4.2` 是一次围绕 `/admin/ai` 工作台继续收口的补丁发布：它把浏览器内 `OpenCode 终端` 正式接进 AI 工作台，同时把正式发版流程从“只推镜像”补成“版本文档 / Git tag / Docker tag / GitHub Release”一致的闭环。

如果只用一句话概括这个版本：

> `v1.4.2` 让管理员可以直接在 `/admin/ai` 打开浏览器终端使用 `opencode`，并把正式发版流程固定成可审计、可回滚、可补 latest 的单一路径。

## 本版摘要

这次补丁主要集中在四个方向：

- `/admin/ai` 新增 `OpenCode 终端` tab，管理员可以在后台直接进入浏览器终端
- 终端直接复用现有 `server` 容器，不新增第六个核心 release 镜像
- AI overlay 启用时，项目目录会挂载到 `/workspace/vanblog`，终端 HOME 会持久化到 `./data/ai-terminal/home`
- 正式发布统一收敛到 `scripts/release-publish.sh` / `pnpm release:publish`，并补齐 `latest` 同步与远端校验

## 相比 v1.4.1 的变化

如果你已经在使用 `v1.4.1` 的 AI 工作台，这一版建议重点关注下面这些变化：

### 1. `/admin/ai` 从两个 tab 变成三个 tab

后台 AI 工作台现在包含：

- 问答页
- 配置中心
- `OpenCode 终端`

终端 tab 本身不会替代原有 AI-QA 配置，也不会自动读取 `/admin/ai` 里填写的 provider / token。`v1.4.2` 的设计目标很明确：先把浏览器终端和预装 CLI 接好，首次 provider/login/模型配置仍然由管理员在终端里自己完成。

### 2. 浏览器终端复用 `server` 容器

这一版没有新增独立 terminal sidecar，也没有再加一个新的 release 镜像。做法是：

- `server` 镜像内预装 `opencode-ai`、`wetty`、`bash`、`git`、`ripgrep`、`python3/pip`、`tmux`
- AI overlay 启用时，由 `server` 同时启动 Nest 服务和 `wetty`
- Caddy 通过 `/admin/ai-terminal/` 专用路由把请求转给 `server:7681`

这样做的结果是：终端和当前部署目录天然对齐，管理员在浏览器里执行 `pwd` 时会直接落在 `/workspace/vanblog`，可以查看当前 compose、脚本、Git 状态和文档。

### 3. 终端鉴权独立于现有 AI-QA cookie / header 逻辑

为了不把终端 iframe 鉴权混进现有问答接口，这一版新增了一套独立接口：

- `GET /api/admin/ai-qa/terminal/status`
- `POST /api/admin/ai-qa/terminal/session`
- `DELETE /api/admin/ai-qa/terminal/session`
- `GET /api/admin/ai-qa/terminal/auth`

管理员先从后台发起会话，再由浏览器带着 path-scoped 的终端 cookie 访问 `/admin/ai-terminal/`。Caddy 会先走 `forward_auth` 验证 cookie，再放行到 `wetty`。这样终端能力只暴露给当前仍具管理员权限的用户，不污染现有后台静态资源路由。

### 4. 发版闭环从“推镜像”升级为“完整版本发布”

`v1.4.2` 还顺手补掉了一个实际踩过的坑：之前可能出现“版本 tag 已推，但 `latest` 未同步成功，导致 `docker-compose.latest.ai.yml` 继续拉旧镜像”的情况。

因此这版之后，正式发版默认统一走：

```bash
pnpm release:publish
```

它会固定完成这三件事：

1. 推送不可变 tag 和版本 tag
2. 把版本 tag 同步成 `latest`
3. 回拉远端镜像，校验 `immutable / version / latest` 三类 tag 的标签一致性

如果版本 tag 已存在但 `latest` 漏推，改用：

```bash
pnpm release:latest
```

而不是再手工逐条 `docker tag` / `docker push`。

## 主要特性 / 修复

### AI 工作台

- `/admin/ai` 新增 `OpenCode 终端` tab
- 终端 tab 支持连接、重新连接和新窗口打开
- 未启用 AI overlay 时，终端 tab 只显示引导说明，不把整页打成错误态

### 终端运行时

- `server` 镜像新增 `wetty` 与常用 CLI
- 终端默认工作目录固定为 `/workspace/vanblog`
- 终端 HOME 持久化到 `./data/ai-terminal/home`
- 不挂 Docker socket，v1 只提供项目目录内的文件、Git、脚本和 `opencode` 使用能力

### 鉴权与路由

- 后端新增独立终端控制器 / provider
- Caddy 为 `/admin/ai-terminal/` 增加专用 `forward_auth` 路由
- 终端 cookie 只作用于 `/admin/ai-terminal/`，避免和后台其他接口混用

### 发布流程

- 根目录新增 `pnpm release:publish`
- 根目录新增 `pnpm release:latest`
- `RELEASE.md` 明确写出：正式发版必须补齐 `package.json` 版本、Git tag、GitHub Release 与 Wiki

## 部署与兼容性说明

`v1.4.2` 没有改变下面这些发布 / 部署边界：

- 默认公开拓扑仍然只有 `caddy`、`server`、`website`、`admin`、`waline`、`postgres`、`redis`
- AI 工作台仍然是按需叠加 `docker-compose.ai-qa.yml` / `docker-compose.fastgpt.yml`
- bundled FastGPT 仍然不是默认公开拓扑的一部分
- `pnpm release:images` / `pnpm release:publish` 仍然只发布 VanBlog 核心 5 个镜像

如果你要使用新的浏览器终端能力，需要满足：

- 通过 `docker-compose.ai-qa.yml` 或 `docker-compose.latest.ai.yml` 启用 AI overlay
- 接受当前部署目录会挂载进 `server` 容器的 `/workspace/vanblog`
- 接受终端 HOME 与 OpenCode 本地配置落盘到 `./data/ai-terminal/home`

## docker compose 配置单

推荐正式部署仍然使用：

- `docker-compose.image.yml`
- `.env.release.example`

核心镜像示例保持：

```env
VANBLOG_DOCKER_REPO=kevinchina/deeplearning
VANBLOG_RELEASE_SUFFIX=v1.4.2-<image-id>
```

主栈启动：

```bash
docker compose -f docker-compose.image.yml up -d
```

只连接已有私有 FastGPT，并启用浏览器终端：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml up -d
```

同时启用 bundled FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml -f docker-compose.fastgpt.yml up -d
```

如果只是 latest 快速体验，也可以继续使用：

```bash
docker compose -f docker-compose.latest.ai.yml up -d
```

## 升级步骤

1. 拉取 `v1.4.2` 对应代码或镜像
2. 确认 `.env` 中的 `VANBLOG_DOCKER_REPO` 与 `VANBLOG_RELEASE_SUFFIX`
3. 执行 `pnpm test:deploy`
4. 执行 `docker compose -f docker-compose.image.yml pull`
5. 执行 `docker compose -f docker-compose.image.yml up -d`
6. 如需 AI 或浏览器终端，再按需叠加 `docker-compose.ai-qa.yml` / `docker-compose.fastgpt.yml`
7. 打开 `/admin` 与 `/admin/ai` 做一次人工验收
8. 如果启用了终端 tab，点击 `连接终端`，确认 `pwd` 为 `/workspace/vanblog`
9. 在终端里确认 `opencode`、`git`、`python3`、`pip`、`rg` 可用

## 回滚方法

如果线上有问题，把 `VANBLOG_RELEASE_SUFFIX` 改回上一版即可，例如：

```env
VANBLOG_RELEASE_SUFFIX=v1.4.1-<image-id>
```

然后重新执行：

```bash
docker compose -f docker-compose.image.yml up -d
```

如果只是浏览器终端能力需要临时下线，也可以先去掉 `docker-compose.ai-qa.yml` / `docker-compose.fastgpt.yml`，继续保留博客主栈在线。
