# v1.4.0

`v1.4.0` 是当前仓库的一次里程碑发布：在保持默认博客部署拓扑不变的前提下，引入了后台 `AI 工作台`、可选的 FastGPT 集成与更明确的文档 / 发布基线。

如果只用一句话概括这个版本：

> `v1.4.0` 把“博客知识问答”正式纳入后台管理能力，但仍坚持 AI 必须按需部署、默认不影响原有博客用户。

## 本版摘要

本次版本主要集中在五个方向：

- 新增后台 `AI 工作台`，入口为 `/admin/ai`，对所有管理员开放
- 管理员问答历史落库并共享，支持继续追问、回看历史、重命名和删除
- 引入 `docker-compose.ai-qa.yml` 与 `docker-compose.fastgpt.yml`，把 AI / FastGPT 明确为可选覆盖层
- 新增 `docker-compose.latest.ai.yml`，提供 one-file latest 快速体验入口
- 把当前发布、部署、测试、AI 运维文档基线统一到 `v1.4.0`

## 相比 v1.3.12 的变化

如果你当前线上或镜像分发仍以 `v1.3.12` 为基线，那么 `v1.4.0` 相比 `v1.3.12` 建议重点关注下面这些变化：

### 1. 后台现在有了正式的 AI 工作台

`v1.3.12` 之前，后台主要围绕文章、草稿、私密文档与移动管理体验收口；`v1.4.0` 则把 AI 工作台提升为一级能力：

- 新增 `/admin/ai` 页面
- 页面拆分为 `博客问答` 与 `配置中心`
- 所有管理员都可以进入，而不是只限超管
- 聊天历史会写入数据库并在管理员之间共享

### 2. AI 回答策略从“封闭拒答”调整为“博客知识优先”

这一版的 AI 策略不是简单的“没引用就不回答”，而是：

- 优先使用已同步到 FastGPT 的博客知识
- 缺少直接覆盖时，可以结合通用知识补充说明
- 系统提示词会要求模型区分“博客里明确写过的内容”和“基于常识的补充判断”

这让 AI 工作台更像一个面向站长的研究助手，而不是只能做严格命中检索的问答框。

### 3. AI / FastGPT 作为可选部署能力被彻底文档化

`v1.4.0` 新增并固化了三类 compose 扩展：

- `docker-compose.ai-qa.yml`：只负责 `server -> FastGPT` 的私有连接注入
- `docker-compose.fastgpt.yml`：负责 bundled FastGPT 及其依赖容器
- `docker-compose.latest.ai.yml`：负责把 latest 主栈和 bundled FastGPT 做成一文件 quick-start

默认 `docker-compose.yml` / `docker-compose.image.yml` 不会被 AI 污染；不需要 AI 的用户，完全可以忽略这几份文件。

### 4. 发布规则明确区分“核心镜像”和“FastGPT 基线”

这一版开始明确约定：

- `pnpm release:images` / `pnpm release:images:push` 只发布 5 个 VanBlog 核心镜像
- AI 工作台代码包含在 `server` / `admin` 核心镜像里
- bundled FastGPT 依赖不属于每次 VanBlog 正式发版的默认推送内容
- FastGPT 当前验证版本矩阵需要单独文档化，并尽量保持稳定，不主动追着上游改

### 5. 文档与测试基线也一起升级到 v1.4.0

这一版把文档口径统一到了当前真实实现：

- 当前版本号更新为 `1.4.0`
- 默认镜像仓库继续固定为 `kevinchina/deeplearning`
- AI 工作台入口统一写成 `/admin/ai`
- latest / image 双轨部署方式写清楚了适用边界
- `pnpm test:deploy`、`pnpm test:blog-flow`、`pnpm test:full` 的职责边界被写清楚

## 主要特性 / 修复

### AI 工作台

- 后台新增 `AI 工作台` 一级菜单
- 支持管理员共享历史会话、继续追问、重命名、删除
- 私密文档索引后的后台跳转地址统一回到 `/admin/editor?...`
- 回答策略改为博客知识优先，并允许受控的通用知识补充

### 可选 FastGPT 部署

- 新增 `docker-compose.ai-qa.yml`
- 新增 `docker-compose.fastgpt.yml`
- 新增 `docker-compose.latest.ai.yml`
- bundled FastGPT 默认只绑定到 `127.0.0.1`
- `fastgpt-bootstrap` 会自动补齐旧数据卷缺失的 `team_subscriptions` 免费套餐记录
- 当前 4 个关键 FastGPT 镜像已经备份到 `kevinchina/deeplearning`，`docker-compose.fastgpt.yml` / `docker-compose.latest.ai.yml` 直接引用这些备份标签

### 文档与发布体系

- README 增补 latest / image 双轨部署说明
- DEPLOY / RELEASE / test-env 文档统一对齐 `/admin/ai`
- 新增独立 AI 运维文档 `docs/ai-qa-fastgpt.md`
- 当前版本说明索引补充 `v1.4.0`

## 部署前注意事项

发布 `v1.4.0` 前，请确认：

- 宿主机 Node 版本满足 `>=24 <25`
- 根目录 `package.json` 版本为 `1.4.0`
- Git tag 使用 `v1.4.0`
- Docker Hub 推送仍以本机 `pnpm release:images:push` 为准
- VanBlog 核心镜像继续发布到 `kevinchina/deeplearning`
- AI / FastGPT 仍然是按需部署能力，不要默认并入所有环境

如果你的站点最近遇到下面任意诉求，这版建议优先升级：

- 想在后台直接围绕博客内容做 AI 问答
- 想把文章、草稿、私密文档同步到一个可维护的知识库问答链路里
- 想把 AI 功能单独部署给部分用户，而不影响其他用户的默认博客部署
- 想固化当前 FastGPT 版本与运维说明，避免以后忘记这套能力是怎么接上的

## docker compose 配置单

推荐生产或分发场景优先使用仓库自带的：

- `docker-compose.image.yml`
- `.env.release.example`

核心镜像仍然使用：

```env
VANBLOG_DOCKER_REPO=kevinchina/deeplearning
VANBLOG_RELEASE_SUFFIX=v1.4.0-<image-id>
```

默认启动博客主栈：

```bash
docker compose -f docker-compose.image.yml up -d
```

如果只连接已有私有 FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml up -d
```

如果同时启用 bundled FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml -f docker-compose.fastgpt.yml up -d
```

如果只是快速体验，也可以使用：

```bash
docker compose -f docker-compose.latest.ai.yml up -d
```

如果你的生产机和我现在一样，只想长期维护两份 quick-start 文件，也可以直接保留：

- `docker-compose.yaml`：不带 AI 的主栈
- `docker-compose.latest.ai.yml`：带 AI 的 one-file 主栈

对应命令：

```bash
# 不启用 AI
docker compose pull
docker compose up -d

# 启用 AI
docker compose -f docker-compose.latest.ai.yml pull
docker compose -f docker-compose.latest.ai.yml up -d
```

当前 bundled FastGPT 版本矩阵见 [`docs/ai-qa-fastgpt.md`](../ai-qa-fastgpt.md)。

## 升级步骤

1. 拉取 `v1.4.0` 对应代码或镜像
2. 确认 `.env` 里的 `VANBLOG_DOCKER_REPO` 与 `VANBLOG_RELEASE_SUFFIX`
3. 执行 `pnpm test:deploy`
4. 执行 `docker compose -f docker-compose.image.yml pull`
5. 执行 `docker compose -f docker-compose.image.yml up -d`
6. 如需 AI，再按需叠加 `docker-compose.ai-qa.yml` / `docker-compose.fastgpt.yml`
7. 打开 `/admin` 与 `/admin/ai` 做一次人工验收

## 回滚方法

如果线上有问题，把 `VANBLOG_RELEASE_SUFFIX` 改回上一版即可，例如：

```env
VANBLOG_RELEASE_SUFFIX=v1.3.12-<image-id>
```

然后重新执行：

```bash
docker compose -f docker-compose.image.yml up -d
```

AI 能力本身也是可选的；如果只是 AI 部分不稳定，也可以先移除 `docker-compose.ai-qa.yml` / `docker-compose.fastgpt.yml`，把博客主栈继续保留在线。
