# v1.3.3

`v1.3.3` 是在 `v1.3.2` 基础上继续收敛后台真实编辑链路的一次补丁版发布。

如果只用一句话概括这个版本：

> `v1.3.3` 重点修复了 `/admin` 中“提示保存成功但实际没有落库”的一批问题，并补齐了 PostgreSQL 模式下 mindmap 与私密文档信息修改的可用性。

## 本版摘要

本次版本主要集中在三个方向：

- 修复后台文章、草稿、文档、动态等编辑保存链路中的假成功问题
- 修复私密文档信息修改与 PostgreSQL 模式下 mindmap 更新流程
- 移除网页版 mindmap 的过时下载提示，并优化镜像构建速度

## 相比 v1.3.2 的变化

如果你当前线上或对外分发仍然以 `v1.3.2` 为基线，那么 `v1.3.3` 相比 `v1.3.2` 可以重点关注下面这些变化：

### 1. `/admin/editor` 的保存成功提示终于和真实结果一致

这一版重点修复了后台编辑页里一类比较隐蔽的问题：前端提示“保存成功”，但后端其实没有真正更新数据。

本次已经覆盖并修复：

- 文章编辑 `/admin/editor?type=article&id=<id>`
- 草稿编辑 `/admin/editor?type=draft&id=<id>`
- 私密文档正文编辑 `/admin/editor?type=document&id=<id>`
- 动态编辑 `/admin/editor?type=moment&id=<id>`

除了后端修复参数处理外，后台前端现在也会校验业务返回值，不再把非成功响应误报成成功。

### 2. 私密文档信息修改不再被错误拦截

此前私密文档在“修改信息”弹窗里，即使只是改标题，也可能把 `library_id` 等位置字段一起提交给后端，触发“文档位置调整请使用移动接口”的保护逻辑，表现为前端提示成功但实际没有生效。

`v1.3.3` 将这条链路收紧为只提交允许修改的字段：

- 私密文档信息弹窗只提交 `title`
- 文案改为“文档标题”
- 不再展示与标题修改无关的字段

这让“改文档标题”与“移动文档位置”重新回到两个独立、可预期的操作路径。

### 3. PostgreSQL 模式下的 mindmap 编辑链路补齐

在 PG / structured-data 模式下，mindmap 之前仍残留部分 Mongo 风格 `_id` 逻辑，导致更新、删除或查看器相关流程不稳定。

`v1.3.3` 已把这部分逻辑切回当前 PostgreSQL 数据结构：

- 更新走 PG 结构化存储
- 删除与查询流程不再依赖 Mongo `_id`
- 配套补上 mindmap provider 的回归测试

如果你的站点已经迁移到 PostgreSQL，这一版比 `v1.3.2` 更值得尽快升级。

### 4. mindmap 网页版过时提示已移除，构建缓存也更合理

这次还顺手处理了两个影响体验的问题：

- 移除了 mindmap 网页版进入编辑页时的“网页版已暂停更新，部分功能缺失，请下载客户端获得完整体验~”提示
- `server` / `waline` Dockerfile 改为使用 BuildKit cache mount，释放 `pnpm` 并行安装能力，减少重复构建时的等待

这不会改变部署拓扑，但会让编辑体验和本机构建体验更顺畅。

## 主要特性 / 修复

### 后台编辑器与信息弹窗

- 修复文章、草稿、文档、动态保存接口的实际落库问题
- 修复前端把后端业务失败误显示为成功的问题
- 修复私密文档“修改信息”只改标题却不生效的问题

### 后端控制器与结构化数据

- 修复多处后台控制器对 `id` 路由参数按数字处理导致的更新失败
- 清理同类问题在 `pipeline`、`collaborator` 等控制器中的实现
- 修复 PostgreSQL 模式下 `mindmap` provider 的更新与删除链路

### 构建与发布体验

- 移除 mindmap 网页版过时下载提示
- 为 `server`、`waline` 镜像增加 pnpm store BuildKit 缓存
- 保持多镜像发布方式不变，继续兼容 `docker-compose.image.yml`

## 部署前注意事项

发布 `v1.3.3` 前，请确认：

- 宿主机 Node 版本满足 `>=24 <25`
- 根目录 `package.json` 版本为 `1.3.3`
- Git tag 使用 `v1.3.3`
- Docker Hub 推送使用本机 `pnpm release:images:push`
- 生产部署优先使用不可变镜像标签，而不是 `latest`

如果你的使用场景依赖后台编辑或私密文档，请在升级后优先验证：

- `/admin/editor?type=article&id=<id>`
- `/admin/editor?type=draft&id=<id>`
- `/admin/editor?type=document&id=<id>`
- `/admin/mindmap`

## docker compose 配置单

推荐生产或分发场景优先使用仓库自带的：

- `docker-compose.image.yml`
- `.env.release.example`

如果你需要给别人一份可以直接改变量部署的示例，下面是一份最小可用 `docker-compose.yaml`。这里直接使用版本别名 tag，不再要求额外填写镜像 id。

```yaml
services:
  caddy:
    image: kevinchina/deeplearning:vanblog-caddy-v1.3.3
    restart: always
    depends_on:
      server:
        condition: service_healthy
      website:
        condition: service_healthy
      admin:
        condition: service_healthy
      waline:
        condition: service_healthy
    environment:
      TZ: Asia/Shanghai
      EMAIL: your-email@example.com
    volumes:
      - ./log:/var/log
      - ./caddy/config:/root/.config/caddy
      - ./caddy/data:/root/.local/share/caddy
    ports:
      - "80:80"

  server:
    image: kevinchina/deeplearning:vanblog-server-v1.3.3
    restart: always
    init: true
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    environment:
      TZ: Asia/Shanghai
      NODE_ENV: production
      VAN_BLOG_DATABASE_URL: postgresql://postgres:postgres@postgres:5432/vanblog
      VAN_BLOG_REDIS_URL: redis://redis:6379
      VAN_BLOG_WALINE_DB: waline
      VAN_BLOG_WALINE_DATABASE_URL: postgresql://postgres:postgres@postgres:5432/waline
      VAN_BLOG_CADDY_MANAGE_HTTPS: "false"
      VANBLOG_CADDY_API_URL: http://caddy:2019
      VANBLOG_WEBSITE_CONTROL_URL: http://website:3011
      VANBLOG_WALINE_CONTROL_URL: http://waline:8361
      VANBLOG_WEBSITE_ISR_BASE: http://website:3001/api/revalidate?path=
      WALINE_JWT_TOKEN: "replace-with-a-strong-random-string"
    volumes:
      - ./data/static:/app/static
      - ./log:/var/log
      - ./aliyunpan/config:/root/.config/aliyunpan
    expose:
      - "3000"
    healthcheck:
      test: ["CMD-SHELL", "wget -q -O /dev/null http://127.0.0.1:3000/swagger || exit 1"]
      interval: 15s
      timeout: 10s
      retries: 10
      start_period: 30s

  website:
    image: kevinchina/deeplearning:vanblog-website-v1.3.3
    restart: always
    init: true
    environment:
      TZ: Asia/Shanghai
      NODE_ENV: production
      PORT: 3001
      WEBSITE_CONTROL_PORT: 3011
      VAN_BLOG_SERVER_URL: http://server:3000
      VAN_BLOG_REVALIDATE: "false"
      VAN_BLOG_ALLOW_DOMAINS: pic.mereith.com
      WALINE_JWT_TOKEN: "replace-with-a-strong-random-string"
    expose:
      - "3001"
      - "3011"
    healthcheck:
      test: ["CMD-SHELL", "wget -q -O - http://127.0.0.1:3011/health >/dev/null || exit 1"]
      interval: 15s
      timeout: 10s
      retries: 10
      start_period: 20s

  admin:
    image: kevinchina/deeplearning:vanblog-admin-v1.3.3
    restart: always
    expose:
      - "3002"
    healthcheck:
      test: ["CMD-SHELL", "wget -q -O /dev/null http://127.0.0.1:3002/admin/ || exit 1"]
      interval: 15s
      timeout: 10s
      retries: 10
      start_period: 20s

  waline:
    image: kevinchina/deeplearning:vanblog-waline-v1.3.3
    restart: always
    init: true
    depends_on:
      postgres:
        condition: service_healthy

      TZ: Asia/Shanghai
      NODE_ENV: production

      PG_PORT: 5432

      PG_USER: postgres
      PG_PASSWORD: postgres
      JWT_TOKEN: "replace-with-a-strong-random-string"
    expose:
      - "8360"
      - "8361"
    healthcheck:
      test: ["CMD-SHELL", "wget -q -O - http://127.0.0.1:8361/health >/dev/null || exit 1"]

    restart: always
    environment:
      TZ: Asia/Shanghai

      POSTGRES_PASSWORD: postgres

      - ./data/postgres:/var/lib/postgresql/data
    healthcheck:

      retries: 10
      start_period: 20s

    restart: always
    volumes:

      interval: 15s
      timeout: 10s
      retries: 10
      start_period: 10s
```
