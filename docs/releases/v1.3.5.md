# v1.3.5

`v1.3.5` 是在 `v1.3.4` 基础上，针对 PostgreSQL 结构化数据链路和后台大列表稳定性做的一次补丁发布。

如果只用一句话概括这个版本：

> `v1.3.5` 重点修复了文章/标签/草稿在 PG 环境下的同步与删除一致性问题，并把后台多个列表页改成轻量查询，避免数据量上来后一次性拉正文导致卡顿。

## 本版摘要

本次版本主要集中在四个方向：

- 修复文章内容为空时的统计崩溃，避免后台列表偶发空白、刷新后恢复
- 修复文章、草稿、标签相关操作在 PG 结构化表中的重复刷新与约束冲突
- 补齐文章维护接口对 PG 结构化表的同步，降低“删除了但列表没及时变化”这类不一致问题
- 优化后台文章、草稿、私密文档、自定义页面列表，不再为列表页额外查询全文正文

## 相比 v1.3.4 的变化

如果你当前线上或外部分发仍以 `v1.3.4` 为基线，那么 `v1.3.5` 相比 `v1.3.4` 可以重点关注下面这些变化：

### 1. 后台文章管理在 PG 环境下更稳定

此前在 PostgreSQL 结构化表模式下，文章管理存在几类容易叠加的问题：

- 某些文章 `content` 缺失时，字数统计会抛异常
- 删除 / 发布 / 标签调整后，后台列表或公开页偶尔不能立即反映最新状态
- 标签同步链路有机会重复触发结构化表重建，进而打到主键或外键约束

`v1.3.5` 把这些链路都做了收敛：

- `wordCount` 对空内容兜底，不再因为异常数据把后台列表打挂
- 文章、草稿、AI 打标、标签刷新改为更明确的异步刷新边界
- 文章结构化表刷新与标签聚合重建增加串行保护，避免并发重建互相踩表

### 2. 文章维护类操作与 PG 同步补齐

这次还补齐了几类以前更像“维护脚本”的操作，让它们在 PG 模式下也能同步刷新结构化表，例如：

- 重排文章 ID
- 清理负数 ID
- 清理临时 ID
- 清理重复 pathname

这样可以避免维护完成后，记录存储层和 PG 查询层短时间内看到不同结果。

### 3. 后台大列表改成轻量查询

这是体感最明显的一项优化。

此前下面这些后台页面虽然是“列表页”，但 PG 查询层仍会把全文字段一并取回：

- `/admin/article`
- `/draft`
- `/document` 的文档树 / 文档库列表
- `/customPage`

当文章、草稿或文档数量比较大时，即使前端只展示标题、分类、时间，后端仍然会多取很多无用正文。

`v1.3.5` 现在改成：

- 列表页默认只查摘要字段
- 详情页 / 编辑页仍然按需查全文
- 后台分页大小与后端上限对齐，避免浏览器残留超大分页值放大卡顿

### 4. 删除与状态变更的一致性更好

这版的另一个重点不是“加新功能”，而是把 PG 模式下的数据读写一致性再往前推一步。

对于已经从旧 Mongo 流程迁移到当前 PG 查询链路的部署，这个版本更适合用来稳定承接生产环境。

## 主要特性 / 修复

### PostgreSQL 结构化数据链路

- `wordCount` 对空内容容错，避免后台管理页因异常文章崩溃
- 标签页刷新与结构化表重建解耦，减少重复重建
- 文章结构化表刷新、标签关联更新、标签聚合重建增加串行保护
- 文章维护类接口补齐 PG 结构化表刷新
- 增加对应 provider / controller / structured-data 回归测试

### 后台列表性能

- 文章列表改成摘要查询，不再为列表页读取文章全文
- 草稿列表改成摘要查询，不再为列表页读取草稿全文
- 私密文档树与文档库列表默认不再读取文档正文
- 自定义页面列表默认不再读取 `html` 正文
- 草稿、文章分页大小收口到后端支持范围内，降低超大分页导致的加载抖动

### 管理端行为修复

- 删除文章后，后台与 PG 查询层状态不同步的问题进一步缓解
- 草稿发布、标签调整后，公开页与后台页刷新边界更清晰
- 后台在 2000+ 文章场景下的列表体验明显轻量化

## 部署前注意事项

发布 `v1.3.5` 前，请确认：

- 宿主机 Node 版本满足 `>=24 <25`
- 根目录 `package.json` 版本为 `1.3.5`
- Git tag 使用 `v1.3.5`
- Docker Hub 推送使用本机 `pnpm release:images:push`
- 生产部署如需精确回滚，仍优先使用不可变镜像标签

如果你的站点已经迁移到 PostgreSQL 查询链路，这版尤其建议同步升级，因为它主要解决的就是 PG 环境下的后台一致性和性能问题。

## 升级步骤

从 `v1.3.4` 升级到 `v1.3.5`，推荐步骤：

1. 备份当前 `data/`、`log/`、`caddy/` 目录
2. 将部署使用的镜像 tag 改到 `v1.3.5` 或对应不可变 tag
3. 执行：

```bash
export VANBLOG_DOCKER_REPO=kevinchina/deeplearning
export VANBLOG_RELEASE_SUFFIX=v1.3.5-<image-id>
docker compose -f docker-compose.image.yml up -d
```

如果你只是想跟随版本别名，也可以把 `VANBLOG_RELEASE_SUFFIX` 设成：

```bash
v1.3.5
```

升级后建议重点自测：

- `/admin/article` 列表加载
- 草稿创建 / 删除 / 发布
- 私密文档树加载与编辑
- 自定义页面列表与详情编辑

## 回滚方法

如果线上有问题，直接把发布后缀改回上一版即可：

```bash
export VANBLOG_DOCKER_REPO=kevinchina/deeplearning
export VANBLOG_RELEASE_SUFFIX=v1.3.4-<image-id>
docker compose -f docker-compose.image.yml up -d
```

如果你使用的是版本别名模式，也可以临时回到：

```bash
export VANBLOG_RELEASE_SUFFIX=v1.3.4
docker compose -f docker-compose.image.yml up -d
```
