# v1.4.1

`v1.4.1` 是一次面向 AI 工作台共享 FastGPT 场景的补丁发布：不改默认部署拓扑、不改 AI 可选覆盖层边界，重点修复“多博客共用一套 FastGPT 时自动资源可能串用”的问题。

如果只用一句话概括这个版本：

> `v1.4.1` 给每个博客补上稳定的 AI 身份标识，并把旧版通用命名的 FastGPT 自动资源平滑迁移到按博客隔离的新命名规则。

## 本版摘要

这次补丁主要集中在四个方向：

- 为每个博客生成并持久化一个只存 PG 的 `blogInstanceId`，作为共享 FastGPT 下的稳定 AI 身份
- VanBlog 自动创建的 `Dataset / App / API Key` 改成 `站点标签 + blogInstanceId` 的 v2 命名规则
- `/admin/ai` 打开时会静默识别旧版 VanBlog 自动资源，并在条件满足时自动迁移到 v2 命名资源
- 后台新增只读身份信息块，明确显示当前博客 UUID、资源管理模式、命名规则版本与自动管理资源名

## 相比 v1.4.0 的变化

如果你已经在使用 `v1.4.0` 的 AI 工作台，并且可能让多套博客共享一套 FastGPT，那么 `v1.4.1` 建议优先关注下面这些变化：

### 1. 每个博客现在都有自己的 AI 身份

`v1.4.0` 的自动资源命名仍然偏“单站默认”；`v1.4.1` 开始，后台会在 AI-QA 配置里持久化一个 `blogInstanceId`（UUID v4）：

- 只保存在当前博客自己的 PostgreSQL 设置中
- 不落宿主机文件，不需要额外挂载目录
- 首次读取 AI-QA 配置时自动生成，后续始终复用
- 如果数据库被整体重建且配置丢失，则会生成新的博客 AI 身份

### 2. 自动资源统一切到 managed v2 命名

以后由 VanBlog 自动创建的 FastGPT 资源会改成下面这套稳定规则：

- `Dataset`：`VanBlog AI 问答知识库 / {siteLabel} / {blogInstanceId}`
- `App`：`VanBlog AI 问答 / {siteLabel} / {blogInstanceId}`
- `API Key`：`VanBlog AI Key / {siteLabel} / {blogInstanceId}`

这里的唯一性不再依赖站点名，而是完全依赖 `blogInstanceId`，因此多个博客共用同一套 FastGPT 时，不会再因为旧版固定名资源而互相串用。

### 3. 旧版自动资源会在后台静默迁移

`v1.4.1` 没有把迁移做成强制弹窗，而是把触发点固定在 `/admin/ai` 页面初始化：

- 页面先正常读取当前配置和状态
- 如果识别到当前博客仍在使用旧版 VanBlog 自动资源，前端会静默调用迁移接口
- 迁移成功后自动切换到新资源并刷新页面状态
- 迁移只处理能够确认是“旧版 VanBlog 自动创建”的资源，不碰手工资源

迁移链路包括：

1. 校验当前资源确实符合旧版 VanBlog 自动资源特征
2. 创建一套 v2 命名的新 `Dataset / App / API Key`
3. 对新 `Dataset` 执行一次全量同步
4. 写回当前博客配置并标记为 `managedV2`
5. 尝试删除旧版自动资源；如果删除失败，只记 warning，不回滚新配置

### 4. 手工配置仍然安全优先

如果管理员手工改了 `datasetId / appId / apiKey`，后台会把资源管理模式切成 `manual`：

- `manual` 模式不会参与静默自动迁移
- 也不会自动删除任何来源不明或人工维护的 FastGPT 资源
- 自动删除逻辑只作用于能确认是旧版 VanBlog 自动创建的资源

## 主要特性 / 修复

### AI 工作台

- 新增 `blogInstanceId` 只读显示
- 新增资源管理模式（`manual` / `managedV2`）只读显示
- 新增当前命名规则版本（v2）只读显示
- 新增当前自动管理资源名只读显示
- 管理员手工改资源 ID 时，自动从托管模式切到 `manual`

### FastGPT 资源隔离

- 自动创建 Dataset / App / API Key 时改用博客级 v2 命名
- 知识同步到 collection metadata 时额外写入 `blogInstanceId`、`resourceNamingVersion: 2`、`managedBy: vanblog`
- 托管资源二次 provision 时会继续沿用当前博客的 UUID 命名，而不是退回旧版固定名

### 迁移与兼容

- 新增 `POST /api/admin/ai-qa/migrate-legacy`
- `/admin/ai` 页面会静默触发旧资源迁移，不打断管理员当前操作
- 缺少 `FASTGPT_ROOT_PASSWORD` 或 bundled 模型配置时，不会做半迁移，页面保持现状
- 旧资源删除失败时，新配置仍保留，只在服务端记录 warning

## 部署与兼容性说明

`v1.4.1` 没有改变下面这些发布 / 部署边界：

- 默认公开拓扑仍然只有 `caddy`、`server`、`website`、`admin`、`waline`、`postgres`、`redis`
- AI 工作台仍然是按需叠加 `docker-compose.ai-qa.yml` / `docker-compose.fastgpt.yml`
- `pnpm release:images` / `pnpm release:images:push` 仍然只发布 VanBlog 核心 5 个镜像
- bundled FastGPT 的固定版本矩阵不在这次补丁里变更

如果你当前已经在使用 `v1.4.0`：

- 单博客 + 单 FastGPT 的场景通常可以直接升级
- 多博客共享 FastGPT 的场景建议优先升级到 `v1.4.1`
- 升级后首次打开 `/admin/ai` 时，建议观察一次状态刷新，确认旧自动资源已切换到带 UUID 的新命名

## docker compose 配置单

推荐正式部署仍然使用：

- `docker-compose.image.yml`
- `.env.release.example`

核心镜像示例保持：

```env
VANBLOG_DOCKER_REPO=kevinchina/deeplearning
VANBLOG_RELEASE_SUFFIX=v1.4.1-<image-id>
```

主栈启动：

```bash
docker compose -f docker-compose.image.yml up -d
```

只连接已有私有 FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml up -d
```

同时启用 bundled FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml -f docker-compose.fastgpt.yml up -d
```

如果只是 latest 快速体验，也可以继续使用：

```bash
docker compose -f docker-compose.latest.ai.yml up -d
```

## 升级步骤

1. 拉取 `v1.4.1` 对应代码或镜像
2. 确认 `.env` 中的 `VANBLOG_DOCKER_REPO` 与 `VANBLOG_RELEASE_SUFFIX`
3. 执行 `pnpm test:deploy`
4. 执行 `docker compose -f docker-compose.image.yml pull`
5. 执行 `docker compose -f docker-compose.image.yml up -d`
6. 如需 AI，再按需叠加 `docker-compose.ai-qa.yml` / `docker-compose.fastgpt.yml`
7. 打开 `/admin` 与 `/admin/ai` 做一次人工验收
8. 如果是共享 FastGPT 场景，首次打开 `/admin/ai` 后确认资源已切换到带博客 UUID 的新命名

## 回滚方法

如果线上有问题，把 `VANBLOG_RELEASE_SUFFIX` 改回上一版即可，例如：

```env
VANBLOG_RELEASE_SUFFIX=v1.4.0-<image-id>
```

然后重新执行：

```bash
docker compose -f docker-compose.image.yml up -d
```

如果只是 AI 工作台共享资源迁移策略需要临时回避，也可以先不叠加 `docker-compose.ai-qa.yml` / `docker-compose.fastgpt.yml`，继续保留博客主栈在线。
