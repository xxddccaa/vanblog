# v1.5.6

`v1.5.6` 是一次围绕后台草稿发布链路的补丁发布，重点是把“草稿箱里的文章无法发布”修到和正式文章创建规则一致。

如果只用一句话概括：

> `v1.5.6` 去掉了草稿发布对 `<!-- more -->` 的后端硬拦截，并补上发布弹窗的真实错误提示，避免“发布失败却提示成功”。

## 本版摘要

这一版主要集中在三件事：

- 修复草稿箱中的草稿因为缺少 `<!-- more -->` 无法发布的问题
- 统一草稿发布和正式文章创建的后端规则，避免同样的内容只能新建文章、却不能从草稿发布
- 收口后台发布弹窗的返回值校验，确保后端失败时前端显示真实错误而不是直接提示成功

## 相比 v1.5.5 的变化

如果你已经在使用 `v1.5.5`，这一版最直接的变化会出现在后台草稿管理和草稿编辑页：

- 没有 `<!-- more -->` 标记的草稿现在也可以正常发布
- 如果草稿发布因为路径名冲突、权限、演示站限制等原因失败，后台会显示接口返回的真实错误
- 发布弹窗的行为现在和后台其他保存/修改弹窗保持一致，不再吞掉后端失败状态

这不是一次功能扩展版本，而是一次针对草稿发布退化的定点修复。

## 主要更新

### 1. 去掉草稿发布对 `<!-- more -->` 的后端硬拦截

这次问题的根因不是草稿内容本身有问题，而是草稿发布链路比正式文章创建链路多了一条额外限制：

- 正式文章创建接口允许没有 `<!-- more -->` 的内容直接保存
- 草稿发布却在后端 `draft.publish()` 中强制要求必须包含 `<!-- more -->`
- 结果同样一篇内容，直接建文章能成功，从草稿发布却被拒绝

`v1.5.6` 现在把这条额外限制去掉，让草稿发布回到和文章创建一致的规则集。

### 2. 修复草稿发布弹窗错误提示

这一版还顺手修了另一个会放大问题体验的点：

- 发布弹窗之前没有像其他后台表单那样检查 `statusCode`
- 后端即使返回失败，前端也可能继续显示“发布成功”

现在发布弹窗会：

- 明确校验接口返回的 `statusCode`
- 非 200 时优先展示后端返回的 `message`
- 只有真正发布成功时才刷新列表和跳转

这样以后如果再遇到路径名冲突、演示站限制或其他发布失败，界面会直接给出真实原因。

### 3. 补充草稿发布回归测试

为了避免后续再次把草稿发布和文章创建规则分叉，这一版补了一条针对草稿发布的后端单测，专门约束：

- 没有 `<!-- more -->` 标记的草稿也可以正常发布成文章

这类测试很小，但对这次问题的回归保护很直接。

## 配置与兼容性

`v1.5.6` 不引入新的环境变量，也不需要数据库迁移：

- 不新增 `.env` 配置项
- 不需要数据库结构迁移
- 不改变 AI / FastGPT 部署方式
- 从 `v1.5.5` 可直接升级

## 发布与部署边界

`v1.5.6` 继续保持当前多镜像发布边界：

- 核心发布物仍然只有 `caddy`、`server`、`website`、`admin`、`waline`
- `postgres` 与 `redis` 继续使用官方镜像
- AI 工作台继续随 `server` / `admin` 核心镜像发布
- bundled FastGPT 仍然是可选叠加，不属于默认主栈公开拓扑

## docker compose 配置单

正式部署仍推荐：

- `docker-compose.image.yml`
- `.env.release.example`

示例环境变量：

```env
VANBLOG_DOCKER_REPO=kevinchina/deeplearning
VANBLOG_RELEASE_SUFFIX=v1.5.6-<image-id>
```

主栈部署：

```bash
docker compose -f docker-compose.image.yml up -d
```

连接已有私有 FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml up -d
```

同时启用 bundled FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml -f docker-compose.fastgpt.yml up -d
```

如果只是 latest 快速体验：

```bash
docker compose -f docker-compose.latest.ai.yml up -d
```

## 升级建议

升级到 `v1.5.6` 后，建议重点看这几个地方：

1. 草稿管理页中的旧草稿，特别是没有 `<!-- more -->` 的草稿，是否已经可以正常发布
2. 如果草稿发布失败，后台是否显示了真实错误，而不是继续提示成功
3. 自定义路径名冲突、演示站限制等已有失败场景，错误提示是否仍然准确
4. 草稿发布后的文章跳转、列表刷新和 ISR 是否保持正常

## 回滚方法

如果线上想回滚到上一版，只需要把：

```env
VANBLOG_RELEASE_SUFFIX=v1.5.5-<image-id>
```

然后重新执行：

```bash
docker compose -f docker-compose.image.yml up -d
```

如果只有后台浏览器表现异常，优先强刷管理端静态资源缓存后再确认。
