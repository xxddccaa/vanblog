# v1.6.6

`v1.6.6` 是一次前台自定义 CSS 注入修复版本，重点解决在 App Router 下自定义 CSS（customCss）被静默丢弃，导致设置背景图后页面上下拉能一直拉到只剩背景图的问题。

如果只用一句话概括：

> `v1.6.6` 让前台自定义 CSS 真正生效，鼠标拖尾画布等依赖自定义 CSS 定位的元素不再把页面撑出整屏空白，配置背景图后滚动到底即为页脚。

## 本版摘要

这一版主要集中在两件事：

- 修复前台 `customCss` 在 Next.js App Router 下被 `next/head` 丢弃的问题，改为直接渲染 `<style>`
- 增加回归测试，按 App Router 真实行为（`next/head` 为空操作）锁定自定义 CSS 必须出现在输出中

## 相比 v1.6.5 的变化

如果你已经在使用 `v1.6.5`，这一版最直接的变化是：

- 后台「自定义 CSS」配置现在会真正注入到前台页面
- 依赖自定义 CSS 的元素（例如鼠标拖尾画布 `.mouse-drag-canvas` 的 `position: fixed`）恢复正常定位，不再以普通流元素撑高页面
- 配置背景图后，页面不再出现额外的整屏空白可滚动区，向下滚动到底即为页脚而非整张背景图

## 主要更新

### 1. 自定义 CSS 注入修复

`packages/website` 使用 Next.js App Router，而 `next/head` 的 `<Head>` 在 App Router 下是空操作，其子节点会被静默丢弃。此前 `CustomLayout` 仅通过 `<Head><style>…</style></Head>` 注入 `customCss`，导致整段自定义 CSS 从未进入页面。

典型表现：用户的自定义脚本创建了 `<canvas class="mouse-drag-canvas">`（鼠标拖尾动画），而把它设为 `position: fixed` 的样式只存在于自定义 CSS 中。样式丢失后，画布以普通流元素留在 `<body>` 末尾，按其 `height`（等于视口高度）撑出整整一屏空白。叠加挂在 `body` 上的固定背景图，这一屏空白就显示成整张背景图——视觉上像「能一直往下拉，最后只剩背景图」。

本版改为直接渲染 `<style>`（以及自定义 head 标签），CSS 不论在 `head` 还是 `body` 都会全局生效：

```tsx
{props.customCss ? <style>{decode(props.customCss)}</style> : null}
```

`customScript` 不受影响，它走 `next/script` 的 `<Script>`，在 App Router 下本就可用。

### 2. 增加回归测试

新增 `packages/website/__tests__/custom-layout-injection.spec.tsx`，按 App Router 的真实行为把 `next/head` mock 成空操作（`() => null`），确保：

- 自定义 CSS 必须出现在 `CustomLayout` 的渲染输出中
- 自定义脚本仍被正常注入

这避免了旧测试把 `next/head` 渲染成 Fragment 从而掩盖问题的情况。

## 配置与兼容性

这一版不引入数据库迁移，不改变 API、环境变量或 compose 拓扑。

如果你之前因为自定义 CSS 不生效而临时把样式搬到别处，可以在本版后改回后台「自定义 CSS」配置。

## docker compose 配置单

如果你使用锁版镜像部署：

```env
VANBLOG_DOCKER_REPO=kevinchina/deeplearning
VANBLOG_RELEASE_SUFFIX=v1.6.6-<image-id>
```

## 升级建议

升级到 `v1.6.6` 后，建议重点验证：

1. 前台已配置背景图时，页面向下滚动到底即为页脚，不再出现整屏只剩背景图的空白
2. 后台「自定义 CSS」中的规则在前台真实生效
3. 鼠标拖尾、心形点击等自定义脚本效果仍正常
4. 暗色 / 亮色模式下背景图显示正常

## 回滚方法

如果线上想回滚到上一版，只需要把：

```env
VANBLOG_RELEASE_SUFFIX=v1.6.6-<image-id>
```

改回上一版的 release suffix，然后重新执行：

```bash
docker compose -f docker-compose.image.yml up -d
```
