# v1.3.6

`v1.3.6` 是在 `v1.3.5` 基础上，针对站点外观配置生效链路和后台 AI 打标可观测性做的一次补丁发布。

如果只用一句话概括这个版本：

> `v1.3.6` 重点修复了前台 favicon 与背景图在当前多容器 / app router 架构下保存后不稳定生效的问题，同时让 AI 打标在上游接口失败时直接显示真实报错，不再只表现为“点了没反应”。

## 本版摘要

本次版本主要集中在三个方向：

- 修复前台站点的 favicon、标题、描述和背景图在 app router 下的同步问题
- 修复 `/favicon.ico` 被 Caddy 改写后仍然 404 的链路，让根路径图标自动跟随后台站点配置
- 修复后台 AI 打标在上游模型接口失败时的错误透传与前端响应校验

## 相比 v1.3.5 的变化

如果你当前线上或外部分发仍以 `v1.3.5` 为基线，那么 `v1.3.6` 相比 `v1.3.5` 可以重点关注下面这些变化：

### 1. 后台站点设置里的 favicon 终于和浏览器默认图标请求打通

此前站点设置中虽然可以保存 `favicon`，前台也会把这个值带到公开接口里，但浏览器默认请求的 `/favicon.ico` 实际上还要经过 Caddy 的特殊改写：

- `/favicon.ico`
- `-> /static/img/favicon.ico`

如果静态目录里没有对应文件，就会得到 404。

`v1.3.6` 为这条链路补了服务端兜底：

- `/favicon.ico` 会重定向到当前 `siteInfo.favicon`
- 被 Caddy 改写后的 `/static/img/favicon.ico` 也会重定向到同一个目标
- 如果站点还没配置 favicon，则回退到 `/admin/logo.svg`

这意味着：

- 浏览器默认图标请求不再和后台设置“各走各路”
- 多容器部署下也不需要再手动同步一个单独的 `favicon.ico` 文件

### 2. 网站背景图、标题和描述在当前前台架构下会稳定同步

这次还修了一个更隐蔽的问题：当前前台已经运行在 app router 上，但站点级 head 信息和背景图变量此前主要还是靠旧的 `next/head` 风格注入。

在某些构建 / hydration 场景下，后台改完站点设置后，真实浏览器里的：

- `document.title`
- `link[rel="icon"]`
- `meta[name="description"]`
- `--bg-image` / `--bg-image-dark`

并不会按预期稳定同步。

`v1.3.6` 现在在前台布局组件里把这些值显式同步到浏览器 DOM，因此：

- 后台“基本设置”里的网站图标会稳定体现在页面 head
- “布局设置”里的背景图和暗色背景图会稳定体现在根元素 CSS 变量
- app router 默认写死标题覆盖动态标题的问题也一并收口

### 3. AI 打标失败时能直接看到真实上游错误

此前 `/admin/site/data?tab=aiTagging` 这页在上游接口失败时，前端没有先校验响应状态，而是直接把 `data` 当作标签字符串去处理。

结果就是：

- 如果模型接口返回 401 / 4xx / 5xx
- 页面往往表现成报错不准确，甚至像“按钮没反应”

`v1.3.6` 现在做了两层修复：

- 后端会优先透传上游 AI 服务返回的 `error.message`
- 前端会先校验 `statusCode`，再决定是否继续解析标签字符串

这样当 API Key 无效、模型名错误、上游服务拒绝请求时，后台就能直接看到更真实的错误提示。

## 主要特性 / 修复

### 前台站点外观

- 修复 `siteInfo.favicon` 在浏览器默认 favicon 请求链路下不生效的问题
- 修复 `/favicon.ico` 与 `/static/img/favicon.ico` 在 Caddy 改写场景下的 404 问题
- 修复站点标题、描述、favicon 在 app router 下的同步问题
- 修复背景图与暗色背景图保存后前台不稳定生效的问题
- 去掉 app router 根布局里写死的默认标题，避免覆盖动态站点标题

### 后台 AI 打标

- AI 打标接口失败时直接显示上游错误信息
- 前端统一校验 `statusCode`，避免把失败响应继续当标签字符串解析
- 单篇打标、批量打标、保存配置、保存标签、触发渲染都补了失败提示
- 补齐 AI 打标 provider 的错误透传测试

## 部署前注意事项

发布 `v1.3.6` 前，请确认：

- 宿主机 Node 版本满足 `>=24 <25`
- 根目录 `package.json` 版本为 `1.3.6`
- Git tag 使用 `v1.3.6`
- Docker Hub 推送使用本机 `pnpm release:images:push`
- 当前正式发布仍默认只推 `linux/amd64`

如果你的站点曾遇到下面任意现象，这版建议优先升级：

- 后台保存了 favicon，但浏览器仍请求 `/favicon.ico` 404
- 后台保存了背景图，但前台一直不更新
- AI 打标点击后提示不准确，无法直接看出是 key / 模型 / 上游接口出错

## docker compose 配置单

推荐生产或分发场景优先使用仓库自带的：

- `docker-compose.image.yml`
- `.env.release.example`

如果你需要给别人一份可以直接改变量部署的示例，下面是一份最小可用 `docker-compose.yaml`。这里直接使用版本别名 tag。

## 回滚建议

如果你在 `v1.3.6` 升级后需要快速回滚：

- 直接把 `docker-compose.image.yml` 或部署环境变量中的版本别名 tag 改回 `v1.3.5`
- 如果你需要严格锁定单次构建结果，则改回对应的不可变 tag
- 回滚后建议手动刷新前台页面缓存，确保 favicon 与背景图等静态外观资源同步回旧版本行为
