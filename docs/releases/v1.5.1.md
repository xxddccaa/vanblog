# v1.5.1

`v1.5.1` 是一次偏后台编辑器默认行为与跨机器部署一致性的补丁版本，重点把“新浏览器 / 新机器没有本地偏好时默认落到 Milkdown”、“实际长期使用者仍然主要依赖 ByteMD” 这类体验偏差收口回稳定状态。

如果只用一句话概括这个版本：

> `v1.5.1` 把后台默认 Markdown 编辑器切回 ByteMD，让新环境、latest 镜像部署和已有运维习惯重新对齐，同时兼容浏览器里已经保存的 ByteMD 本地记录。

## 本版摘要

这次正式版本主要集中在三个方向：

- 当浏览器里还没有保存 `vanblog_editor_engine` 时，后台默认编辑器重新回到 ByteMD
- 已经把 `vanblog_editor_engine` 记录成 `bytemd` 的浏览器，升级后行为保持不变
- 发布文档、部署示例和自动化校验统一推进到 `v1.5.1`，让镜像锁版、latest 同步和 GitHub Release 口径保持一致

## 相比 v1.5.0 的变化

如果你已经在使用 `v1.5.0`，并且当前浏览器本地已经把 `vanblog_editor_engine` 记成 `bytemd`，那升级到 `v1.5.1` 后通常不会感觉到行为变化。

这一版真正收口的是“没有历史偏好”的场景，例如：

- 新机器首次登录后台
- 新浏览器或无痕窗口首次打开编辑页
- 清空站点本地存储后重新进入编辑器
- 通过 `latest` 镜像在另一台机器上新部署并首次验收后台

这些情况下，`v1.5.1` 现在会把默认值重新落到 ByteMD，而不是继续把无效值或空值兜底成 Milkdown。

### 1. 默认编辑器回到 ByteMD

这次补丁把后台编辑器的默认解析逻辑收回到更符合现有使用习惯的路径：

- `vanblog_editor_engine` 缺失时，默认返回 ByteMD
- `vanblog_editor_engine` 是无效值时，同样回退到 ByteMD
- `vanblog_editor_engine` 已经写成 `bytemd` 时，继续沿用当前浏览器的既有记录

这让“新环境默认行为”和“当前运维、写作习惯”重新保持一致。

### 2. 已存在的 ByteMD 本地记录继续兼容

`v1.5.1` 不是增加一套新的迁移流程，而是把“默认值”改回 ByteMD。

也就是说：

- 如果当前浏览器本地已经保存过 `vanblog_editor_engine=bytemd`，升级后仍然保持原样
- 这次升级不会清理已有浏览器缓存
- 想人工确认默认行为时，可以先清掉 `localStorage` 里的 `vanblog_editor_engine` 再重新打开编辑页

因此这个版本属于“默认行为对齐”，不是“额外增加编辑器迁移动作”。

### 3. 镜像部署与文档基线同步推进到 v1.5.1

除了代码本身，这一版也把正式发版相关口径统一推进到 `v1.5.1`：

- 根目录版本号切到 `1.5.1`
- `README.md`、`RELEASE.md`、`DEPLOY.md` 的示例标签和版本说明同步更新
- 发布说明索引与部署校验测试一起指向新的 `docs/releases/v1.5.1.md`

这样后续执行 `pnpm release:publish`、锁版部署或 GitHub Release 时，不会继续混用上一版文案。

## 主要特性 / 修复

### 后台编辑器默认行为

- 默认 Markdown 编辑器重新回到 ByteMD
- 空值 / 非法值统一按 ByteMD 兜底
- 新浏览器、新机器和无痕窗口的首次进入体验更稳定

### 本地偏好兼容性

- 继续保留 `vanblog_editor_engine` 本地存储键
- 已有 `bytemd` 本地记录继续兼容
- 不引入新的环境变量或迁移步骤

### 发布与部署口径

- 仓库版本基线推进到 `v1.5.1`
- 发布说明索引补充 `v1.5.1` 条目
- 自动化部署文档校验同步覆盖新版本说明

## 部署与兼容性说明

`v1.5.1` 没有改变下面这些发布 / 部署边界：

- 默认公开拓扑仍然只有 `caddy`、`server`、`website`、`admin`、`waline`、`postgres`、`redis`
- AI 工作台仍然按需叠加 `docker-compose.ai-qa.yml` / `docker-compose.fastgpt.yml`
- bundled FastGPT 仍然不是默认公开拓扑的一部分
- `pnpm release:images` / `pnpm release:publish` 仍然只发布 VanBlog 核心 5 个镜像

这意味着：

- 不需要修改现有 compose 编排
- 不需要新增环境变量
- 已经在使用 `v1.5.0` 的部署可以直接升级到 `v1.5.1`

## docker compose 配置单

推荐正式部署仍然使用：

- `docker-compose.image.yml`
- `.env.release.example`

核心镜像示例保持：

```env
VANBLOG_DOCKER_REPO=kevinchina/deeplearning
VANBLOG_RELEASE_SUFFIX=v1.5.1-<image-id>
```

主栈启动：

```bash
docker compose -f docker-compose.image.yml up -d
```

只连接已有私有 FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml up -d
```

同时启用 bundled FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml -f docker-compose.fastgpt.yml up -d
```

如果只是 latest 快速体验，也可以继续使用：

```bash
docker compose -f docker-compose.latest.ai.yml up -d
```

## 升级步骤

1. 拉取 `v1.5.1` 对应代码或镜像
2. 确认 `.env` 中的 `VANBLOG_DOCKER_REPO` 与 `VANBLOG_RELEASE_SUFFIX`
3. 执行 `pnpm test:deploy`
4. 执行 `docker compose -f docker-compose.image.yml pull`
5. 执行 `docker compose -f docker-compose.image.yml up -d`
6. 打开 `/admin/editor`，先用无痕窗口或清理 `vanblog_editor_engine` 后做一次人工验收
7. 确认默认打开的是 ByteMD，而不是依赖旧浏览器缓存才看到熟悉编辑器
8. 如需重复验收默认行为，先确认当前浏览器没有残留旧的 `vanblog_editor_engine` 本地值

## 回滚方法

如果线上有问题，把 `VANBLOG_RELEASE_SUFFIX` 改回上一版即可，例如：

```env
VANBLOG_RELEASE_SUFFIX=v1.5.0-<image-id>
```

然后重新执行：

```bash
docker compose -f docker-compose.image.yml up -d
```

如果只是某个浏览器残留了旧的本地状态，也可以先清理站点存储后重新验收，不影响 AI overlay 与 `/admin/ai` 的现有部署方式。
