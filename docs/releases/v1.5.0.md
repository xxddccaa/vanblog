# v1.5.0

`v1.5.0` 是一次偏后台初始化与发布稳定性的正式版本，重点把“已初始化站点误回到 `/init`”、“编辑器插入 emoji 时光标位置错乱”、以及“隐藏 / 私密文章更新后 ISR 失效不稳定”这几类容易影响真实使用的细节一起收口。

如果只用一句话概括这个版本：

> `v1.5.0` 让后台初始化入口、文章增量刷新和编辑器输入体验更稳，避免管理员在已经上线的站点里走到错误流程，也避免隐藏内容的缓存刷新因为查询路径或异步异常而拖垮正常保存链路。

## 本版摘要

这次正式版本主要集中在四个方向：

- 初始化页现在会先检查站点是否已经完成初始化，避免已上线站点继续进入首次安装流程
- ByteMD 表情选择器会记住上一次光标位置，插入 emoji 时不再跑到错误位置
- 文章创建 / 更新 / 删除后的 ISR 触发改成异步兜底，隐藏或私密文章也能走正确的刷新查询路径
- 机器本地测试环境文档同步切到新的 `/data/xiedong/test-vanblog` 目录与 `18083` 端口，减少后续 AI / 人工排障时继续使用旧路径

## 相比 v1.4.3 的变化

如果你已经在使用 `v1.4.3` 的后台登录过期跳转链路，这一版不改变既有部署拓扑，也不增加新的核心镜像；它主要补的是后台初始化、编辑器和 ISR 刷新这些更偏“日常维护稳定性”的问题。

### 1. 已初始化站点不再误进 `/init`

之前如果用户或管理员直接打开 `/init`，前端页面本身不会先判断当前站点是否已经初始化完成，这会带来两个问题：

- 已上线站点仍可能看到首次初始化向导
- 登录态和未登录态在这个页面上的跳转不够明确

`v1.5.0` 现在做了两层收口：

- 后台新增 `GET /api/admin/init/check`，专门返回当前站点是否已经初始化
- 管理后台初始化阶段与 `/init` 页面都会先请求这个接口

因此现在的行为会更清晰：

- 已初始化 + 已登录：直接回到后台首页
- 已初始化 + 未登录：直接跳到登录页
- 只有未初始化站点，才会继续显示初始化向导

### 2. 编辑器插入 emoji 时保留光标位置

这一版修掉了 ByteMD 表情面板的一个老问题：

- 打开 emoji 面板时记住当前选区
- 选择 emoji 后先恢复选区，再插入字符
- 插入后继续把焦点留在编辑器里

这样管理员在正文中间插入表情时，不会再出现“明明点在 A 段，结果 emoji 跑到文末或别的位置”的错位体验。

### 3. 隐藏 / 私密文章的 ISR 刷新改走正确查询路径

过去文章增量刷新在创建或更新时会走公开访问路径去取文章详情，这对普通公开文章没有问题，但对隐藏或私密文章不够稳。

`v1.5.0` 把这里收成了后台查询语义：

- 创建 / 更新事件直接按文章 id 查询
- 不再依赖只适合公开读取链路的 `getByIdOrPathnameWithPreNext`
- 对没有公开 pathname 的隐藏文章，也能稳定刷新 `/post/<id>` 等相关页面

这一版也补了对应测试，避免以后再把隐藏内容误带回公开查询逻辑。

### 4. ISR 异步失败不再拖垮文章保存 / 删除接口

文章编辑完成后，ISR 和 AI 问答同步本来就属于“副作用链路”。

`v1.5.0` 现在会把文章相关 ISR 触发统一包成异步兜底：

- 即使某次增量刷新失败，也只记录日志
- 文章创建、更新、删除接口本身仍然返回成功结果
- 避免因为缓存刷新抖动，把后台正常保存误表现成接口失败

## 主要特性 / 修复

### 后台初始化链路

- 新增 `GET /api/admin/init/check`
- `/init` 页面首次渲染时先检查初始化状态
- 已初始化站点会按是否已有 token 自动跳回后台首页或登录页

### 编辑器体验

- emoji 面板插入前保存当前选区
- 选中 emoji 后恢复选区并聚焦编辑器
- 减少正文中间编辑时的插入错位

### ISR 与缓存刷新

- 文章 ISR 触发增加异步异常兜底
- 隐藏 / 私密文章刷新改用按 id 的后台查询
- 补齐隐藏文章刷新和 ISR 失败不影响主流程的自动化测试

### 本机测试环境文档

- `docs/reference/test-env.md` 改为记录当前机器上的新测试目录
- 统一说明 `18083` 入口与 `/data/xiedong/test-vanblog` 的实际工作方式
- 明确前端改动需要先在源码目录重新构建再去测试目录重建容器

## 部署与兼容性说明

`v1.5.0` 没有改变下面这些发布 / 部署边界：

- 默认公开拓扑仍然只有 `caddy`、`server`、`website`、`admin`、`waline`、`postgres`、`redis`
- AI 工作台仍然按需叠加 `docker-compose.ai-qa.yml` / `docker-compose.fastgpt.yml`
- bundled FastGPT 仍然不是默认公开拓扑的一部分
- `pnpm release:images` / `pnpm release:publish` 仍然只发布 VanBlog 核心 5 个镜像

这意味着：

- 不需要修改现有 compose 编排
- 不需要新增环境变量
- 已经在使用 `v1.4.3` 的部署可以直接升级到 `v1.5.0`

## docker compose 配置单

推荐正式部署仍然使用：

- `docker-compose.image.yml`
- `.env.release.example`

核心镜像示例保持：

```env
VANBLOG_DOCKER_REPO=kevinchina/deeplearning
VANBLOG_RELEASE_SUFFIX=v1.5.0-<image-id>
```

主栈启动：

```bash
docker compose -f docker-compose.image.yml up -d
```

只连接已有私有 FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml up -d
```

同时启用 bundled FastGPT：

```bash
docker compose -f docker-compose.image.yml -f docker-compose.ai-qa.yml -f docker-compose.fastgpt.yml up -d
```

如果只是 latest 快速体验，也可以继续使用：

```bash
docker compose -f docker-compose.latest.ai.yml up -d
```

## 升级步骤

1. 拉取 `v1.5.0` 对应代码或镜像
2. 确认 `.env` 中的 `VANBLOG_DOCKER_REPO` 与 `VANBLOG_RELEASE_SUFFIX`
3. 执行 `pnpm test:deploy`
4. 执行 `docker compose -f docker-compose.image.yml pull`
5. 执行 `docker compose -f docker-compose.image.yml up -d`
6. 打开 `/admin` 和 `/init` 做一次人工验收，确认已初始化站点不会再进入初始化向导
7. 在后台编辑一篇公开文章和一篇隐藏文章，确认保存后前台页面与缓存刷新正常
8. 在正文中间插入 emoji，确认插入位置和焦点行为正确

## 回滚方法

如果线上有问题，把 `VANBLOG_RELEASE_SUFFIX` 改回上一版即可，例如：

```env
VANBLOG_RELEASE_SUFFIX=v1.4.3-<image-id>
```

然后重新执行：

```bash
docker compose -f docker-compose.image.yml up -d
```

如果只是本机测试环境文档或初始化入口体验需要临时回退，也不影响 AI overlay 与 `/admin/ai` 的现有部署方式。
