# v1.3.0

`v1.3.0` 是这次仓库“大升级”的收敛版本，核心目标很明确：

- 把全仓运行时基线升级到 `Node.js 24`
- 保留当前多镜像部署拓扑不变
- 修复升级后后台管理在主题、布局、日志、调试链路上的一批兼容问题

如果只用一句话概括这个版本：

> `v1.3.0` 主要是 Node 版本升级，但不是只改 `package.json`，而是把宿主机开发、Docker Compose、后台兼容和发布流程一起补齐了。

## 这版为什么重要

在 `v1.2.1` 之后，仓库的缓存和镜像部署能力已经比较稳定，但开发运行时仍然偏旧。

这会带来几个问题：

- 宿主机开发环境与镜像环境不完全一致
- 本地调试时，Node 版本一旦变动，后台和评论等子系统容易出现兼容问题
- 文档里缺少明确的“宿主机调试”和“镜像态调试”双路径说明
- 后台一些界面在升级后会出现暗色模式、布局、日志读取等异常

`v1.3.0` 就是用来把这些问题一次性收口的版本。

## 核心变化

### 1. 运行时基线升级到 Node 24

当前仓库已经统一到：

- `Node.js 24.14.1`
- `pnpm 10.33.0`

对应调整包括：

- 根目录 `package.json` 的 engines 已更新到 `>=24 <25`
- 新增 `.nvmrc`
- 新增 `.node-version`
- Dockerfile 与多镜像构建链路统一切到 Node 24 基线
- CI / 本地发布 / 宿主机调试流程同步对齐

这意味着：

- 宿主机开发更容易与镜像环境保持一致
- 后续继续做大版本依赖升级时，基础运行时不会再成为主要阻塞点

### 2. 宿主机调试链路正式成型

这版把“宿主机直接跑”和“Docker Compose 镜像态验收”都整理成了明确流程。

新增和补齐的内容包括：

- `scripts/host-dev-up.sh`
- `scripts/host-dev-down.sh`
- `scripts/host-dev-status.sh`
- `scripts/host-dev-env.sh`
- `tests/host-dev/docker-compose.yaml`
- `tests/host-dev/Caddyfile`
- `docs/host-debug.md`

现在推荐的调试方式是：

- **快速迭代改代码**：优先用宿主机 `18080`
- **最终验收发布行为**：回到 Docker Compose `18080`

这样既保留了宿主机热更新的速度，也不会丢掉镜像态验证。

### 3. 后台兼容性与体验修复

这次 Node 升级过程中，后台管理暴露出一批升级后问题，这版已经一并修复，包括：

- `/admin` 子路径下的运行时兼容修复
- 登录与运行时初始化问题修复
- 左下角主题切换 / 登出入口恢复
- 亮色 / 暗色主题不一致问题修复
- 文档树、评论管理、系统设置等多个页面的暗色背景修复
- 系统日志页无法加载、一直转圈的问题修复

尤其是系统日志这块，这次做了两件关键改动：

- 后端启动时会把 `stdout/stderr` 镜像到 `vanblog-stdio.log`
- 日志读取逻辑不再依赖固定不存在的路径，并支持回退与异常兜底

所以现在：

- 宿主机 dev 能看到系统日志
- Docker Compose 里也能正常看到系统日志

### 4. 前台与构建链路的现代化调整

这版还顺手推进了前台和构建的一些基础整理：

- website 逐步迁到 App Router 结构
- SSR / 公共页面 shell 与缓存读取逻辑继续整理
- 发布测试与当前栈测试补充
- Docker / 文档 / AI 指南同步更新
- GitHub 侧 release workflow 移除，Docker Hub 发布统一回到本机手工推送

这些变化虽然不一定直接体现在“首页看起来完全变了”，但它们会让后续继续升级 React、Next、后台依赖时更稳。

## 推荐升级方式

如果你从 `v1.2.1` 升到 `v1.3.0`，建议这样做：

1. 先把宿主机 Node 切到 `24.14.1`
2. 执行 `pnpm install`
3. 运行 `pnpm test:full`
4. 需要本地调试时，用 `pnpm host:dev:up`
5. 需要镜像态验收时，用 `docker compose up -d --build`

如果你是生产部署，仍然推荐优先使用镜像部署：

- `docker-compose.image.yml`
- `.env.release.example`
- 不可变发布 tag：`v1.3.0-<image-id>`

## 适合谁升级

如果你现在的目标是下面这些之一，那么 `v1.3.0` 值得升级：

- 想把博客开发环境统一到 Node 24
- 想让宿主机开发和 Docker 环境更一致
- 想减少后台在新运行时下的兼容问题
- 想给后续 React / Ant Design / 生态升级先打好运行时基础

如果你期待的是“只升级 Node，别动别的”，那这版也适合你，因为它的主线确实就是 Node 24；只是为了让这个升级可用、可测、可发布，顺手把必要的兼容坑一起填平了。
