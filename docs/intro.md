VanBlog 是一套面向个人博客场景的完整系统，包含公开站点、后台管理、后端 API、评论服务，以及一套适合长期维护的 Docker Compose 部署方案。

<!-- more -->

## 当前仓库的定位

这个仓库最初源于 VanBlog，但现在已经以当前实现为主线持续维护。你在这里看到的部署方式、文档说明、镜像发布和回滚流程，都以本仓库为准，而不是旧版单容器方案。

## 当前默认架构

当前默认部署会拆分成以下服务：

- `caddy`：对外统一入口，负责 `/`、`/admin`、`/api`、评论等路由
- `server`：NestJS API、静态资源、Swagger
- `website`：Next.js 前台站点
- `admin`：后台静态资源，统一挂到 `/admin/`
- `waline`：独立评论服务
- `postgres`：主站与评论共用的 PostgreSQL 实例，仅在内部网络访问
- `redis`：缓存，仅在内部网络访问

这样做的主要好处是：

- 服务边界清晰，便于排障和独立重启
- 后台不再依赖单独暴露 `3002`，统一走 `/admin`
- `postgres`、`redis` 与 Caddy admin API 默认不对外暴露，安全边界更明确
- 生产环境可以直接基于多镜像版本做回滚与追踪

## 主要特性

- [x] 前台、后台、后端与评论服务完整集成
- [x] Docker Compose 多容器部署，支持源码构建和已发布镜像部署
- [x] 自动 HTTPS、反向代理、`/admin` 子路径访问
- [x] 草稿、分类、标签、搜索、TOC、评论、友链、打赏、自定义导航栏
- [x] Markdown 编辑器，支持 Mermaid、LaTeX、代码高亮块、图片上传
- [x] 图床支持本地存储与多种第三方存储
- [x] SSG + ISR 风格前台生成与更新能力
- [x] 支持协作者、Token、日志、RSS、导入导出与 API 调用
- [x] 提供部署测试、博客流程测试与多镜像发布文档

## 适合什么场景

VanBlog 更适合这些需求：

- 想要一套可直接上线的个人博客系统
- 既要后台写作体验，也要前台性能与 SEO
- 希望部署结构清晰，可独立查看日志和重启服务
- 希望后续可以持续做功能定制，而不是受限于单一镜像黑盒

## 当前维护重点

当前仓库更关注以下方向：

- 优化前台分类、标签、导航与阅读体验
- 优化后台列表、分页、搜索与管理流程
- 继续补齐多容器部署下的文档、测试和发布规范
- 让日常维护、升级、回滚都尽量标准化
