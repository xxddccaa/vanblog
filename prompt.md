# VanBlog 架构与性能改造任务契约

## 定义

这是一份任务契约，而不是普通备注。
你必须把“目标”视为最终交付标准，把“执行步骤”视为可调整的施工计划。
允许优化步骤、顺序和方法，但不允许擅自删除、弱化或改写目标。
只有当所有目标都已实际完成、相关修改已落盘、并完成必要自检后，任务才算完成。
不要把“已分析”“已思考”“已给建议”“列了方案”视为完成，必须以实际交付结果为准。

## 背景

- 当前项目是一个基于 VanBlog 的 monorepo，核心目录包括 `packages/server`、`packages/website`、`packages/admin`、`packages/waline`。
- 该任务最初是为了解决“单容器 + Caddy 糅合运行”带来的调试与回归困难；仓库当前已经落地为多容器 compose 架构，但这份契约保留为改造记录与验收基线。
- 当前性能体验存在明显瓶颈，尤其是博客首页、标签页、分类/时间线等公共读路径存在全量加载、重复查询、接口过重或前端渲染过重的问题。
- 当前缺少足够的自动化测试，很多改动依赖手工验证，风险高、回归成本大。
- 本次任务允许进行架构级调整，不局限于当前技术栈；在有明确收益时，可以引入 Go、Redis、PostgreSQL 等实现，但必须真正落地到仓库，而不是停留在建议层面。

## 当前落地状态（2026-04）

- 部署已拆分为 `caddy`、`server`、`website`、`admin`、`waline`、`mongo` 六个服务，由根目录 `docker-compose.yml` 编排。
- `mongo` 与 Caddy admin API `2019` 默认不对外暴露，后台统一通过 `/admin` 子路径访问。
- 仓库已提供 `pnpm test:deploy` 与 `pnpm test:blog-flow`，用于部署配置检查与整站博客流程回归。
- 后台静态资源、服务间控制通信、前台 SSR/SSG 基础地址已经按多容器模型调整。

## 工作边界

- 允许修改的范围：
  - `packages/`
  - `docker/`
  - `docker-compose.yml`
  - `scripts/`
  - `docs/`
  - `README.md`
  - 测试目录与测试配置
  - 新增为本任务服务的目录，例如 `services/`、`infra/`
- 不允许做的事情：
  - 只输出方案，不落地代码
  - 为了省事而删除核心功能
  - 未说明原因就引入重量级依赖
  - 未提供运行方式就新增服务
- 允许新增依赖、缓存层、存储层或新服务，但必须满足：
  - 有明确目标收益
  - 有对应代码、配置、文档
  - 有最小可运行方式
  - 有兼容或迁移说明
- 保持现有业务能力可用；如果需要调整现有行为，必须在结果中明确说明影响范围。

## 不可变目标

- 为关键路径建立可运行的自动化测试，至少覆盖首页文章列表加载逻辑、公共元数据接口、标签或分类查询/分页逻辑，并在仓库中提供统一测试入口与使用说明
- 将当前单容器部署拆分为多容器架构，至少实现 `server`、`website`、`admin` 的独立构建与独立运行，并提供可直接启动的本地开发或调试编排配置
- 完成公共读路径的性能改造，使首页/分页列表不再加载整篇文章正文，标签页不再一次性拉取全部标签数据，且至少再解决一个高频页面或接口的明显性能瓶颈
- 对本次改造涉及的架构与技术选型做出可运行落地；如引入 Redis、PostgreSQL、Go 或其他组件，必须在仓库中提供实际接入、配置方式、用途说明与收益说明；如最终未引入，也必须明确说明为什么现状下不需要
- 补齐与本次改造对应的文档，至少包括运行方式、测试方式、服务拆分说明、性能优化点说明，以及新增组件的启停或配置方法

## 执行步骤

- 先阅读并梳理当前项目的启动链路、接口读路径、首页/标签/分类/时间线页面的数据来源与渲染方式
- 先建立最小可运行的测试基线，再进行核心重构，避免改完只能手测
- 优先解决影响最大的性能问题：请求体过大、全量拉取、重复查询、SSR/SSG 期间无缓存、列表页重型渲染
- 设计并落地多容器方案时，优先保证本地调试体验：服务可单独启动、单独重启、单独查看日志
- 如需引入 Redis、PostgreSQL、Go 等新组件，先明确它解决的是哪个实际瓶颈，再落地实现与文档
- 保持目标不变，但允许根据代码现状调整实施顺序与技术方案

## 输出要求

- 修改必须直接落盘到仓库，而不是只给建议
- 说明中必须给出关键文件路径
- 必须说明新增的运行命令、测试命令、容器启动方式
- 如果做了技术取舍，必须说明原因
- 如果某项引入了新组件，必须说明它解决了什么问题、为什么值得引入

## 验收标准

- 所有不可变目标都能逐项对账
- 自动化测试可以被明确命令触发，且覆盖本次改造的关键行为
- 多容器方案可用于本地启动和调试，服务边界清晰
- 首页与其它高频公共页面/接口的优化不是口头说明，而是实际代码与配置改动
- 文档中的命令、端口、路径、组件说明与仓库实际实现一致
- 不引入明显的构建、运行或依赖错误

## 补充说明

- 如果发现当前实现与目标冲突，以目标为准，做最小必要重构
- 可以分阶段推进，但不得把“阶段性完成”当成“任务完成”
- 如果某项目标因外部条件阻塞，必须明确说明阻塞点、已完成部分、未完成部分，以及还差什么才能完成
- 默认优先选择长期可维护、可测试、可调试的实现，而不是临时补丁
